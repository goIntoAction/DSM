### 1. 生物识别启动认证问题

**现象**: 应用启动时生物识别对话框显示不正常，有时无法弹出或导致崩溃。

**原因**:
- 生物识别对话框需要在 Activity 完全准备好后才能显示
- 在 `onCreate` 中直接调用会导致 Window 未准备好

**解决方案**:

```kotlin
// MainActivity.kt
override fun onResume() {
    super.onResume()
    if (!hasCheckedAuth) {
        hasCheckedAuth = true
        // 延迟调用，确保 Activity 完全准备好
        window.decorView.post {
            checkLaunchAuth()
        }
    }
}
```

**经验**:
- 生物识别对话框必须在 UI 完全加载后显示
- 使用 `window.decorView.post` 延迟执行
- 添加 `hasCheckedAuth` 标志避免重复检查

**相关 Commit**: `8b0a16b fix: 修复启动生物识别认证功能`

### 2. CI 编译错误 - 签名配置问题

**现象**: GitHub Actions CI 编译失败，报签名相关错误。

**原因**:
- 签名配置文件在 keystore 不存在时仍然尝试创建签名配置
- 导致 Gradle 无法找到签名文件

**解决方案**:

```kotlin
// app/build.gradle.kts
signingConfigs {
    val storeFilePath = System.getenv("RELEASE_STORE_FILE") ?: findProperty("RELEASE_STORE_FILE") as? String
    if (storeFilePath != null && storeFilePath.isNotEmpty() && file(storeFilePath).exists()) {
        create("release") {
            storeFile = file(storeFilePath)
            // ... 其他配置
        }
    }
}

buildTypes {
    release {
        // 如果签名配置存在则使用，否则使用默认调试签名
        signingConfig = signingConfigs.findByName("release") ?: signingConfigs.getByName("debug")
    }
}
```

**经验**:
- 签名配置应该条件性地创建
- 始终提供降级方案（调试签名）

**相关 Commit**: `6032f1b fix: 修复 build.gradle.kts 签名配置编译错误`

### 3. MPV 播放器库依赖冲突

**现象**: 编译时出现 `libc++_shared.so` 冲突错误。

**原因**:
- MPV 库和 termlib 都依赖 `libc++_shared.so`
- 不同版本的 NDK 编译的库可能不兼容

**解决方案**:

```kotlin
// app/build.gradle.kts
packaging {
    jniLibs {
        useLegacyPackaging = true
        pickFirsts += setOf(
            "lib/arm64-v8a/libc++_shared.so",
            "lib/armeabi-v7a/libc++_shared.so",
            "lib/x86/libc++_shared.so",
            "lib/x86_64/libc++_shared.so"
        )
    }
}
```

**经验**:
- 使用 `pickFirsts` 选择第一个遇到的库
- 确保 MPV 库在依赖树中排在前面

**相关 Commit**: `b73abde docs: 修正 mpv-android 库链接为实际使用的 fork`

### 4. 多架构 APK 编译配置

**现象**: Release 包只生成一个通用 APK，体积过大。

**原因**:
- 未配置 ABI 分离编译

**解决方案**:

```kotlin
splits {
    abi {
        isEnable = true
        reset()
        include("armeabi-v7a", "arm64-v8a", "x86", "x86_64")
        isUniversalApk = true // 同时生成通用 APK
    }
}
```

**经验**:
- 启用 ABI 分离减小编译体积
- 保留 universal APK 方便分发

## 重要决策记录

### 1. 选择 MVI 架构

**背景**:
- 需要清晰的单向数据流
- 避免 MVVM 中常见的状态管理混乱

**可选方案**:
- MVVM (传统 Android 方案)
- MVI (单向数据流)
- Clean Architecture (复杂度高)

**最终决策**: MVI + Repository 模式

**原因**:
- MVI 提供清晰的单向数据流
- Intent/State/Event 分离易于测试
- Repository 模式统一数据访问

**经验**:
- MVI 适合网络密集型应用
- ViewModel 基类简化代码复用

### 2. 使用 Retrofit 替代手动 HTTP

**背景**:
- 初期使用手动 OkHttp 调用
- 代码重复，难以维护

**最终决策**: 迁移到 Retrofit

**原因**:
- 类型安全的 API 定义
- 自动 JSON 解析
- 更好的错误处理

### 3. 多 OkHttpClient 策略

**背景**:
- 文件下载和普通 API 共用一个客户端
- 导致文件下载时阻塞其他请求

**最终决策**: 创建三个专用客户端

```kotlin
lateinit var okHttpClient: OkHttpClient           // 普通 API（60s 超时，带重试）
lateinit var fileTransferClient: OkHttpClient     // 文件上传/下载（300s 超时，无重试）
lateinit var imageClient: OkHttpClient            // 图片加载（60s 超时，无重试）
```

**原因**:
- 文件传输需要更长超时
- 图片加载不需要重试
- 共享连接池提高效率
